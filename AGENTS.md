
# 铁律：防Git冲突 — 改代码前先同步

**改任何源码之前，必须：**
1. `git pull` — 拉取最新代码，看对方有没有改东西
2. 读 `FRIDAY.md` — 看对方改了什么文件
3. 如果对方正在改同一个文件，先沟通，不要覆盖

**改完任何源码之后，必须：**
1. 在 `FRIDAY.md` 的"最近改动"段写下：改了什么文件、什么内容、为什么
2. `git push` — 推送让对方可见

**严禁：**
- 不 pull 就直接改代码
- 覆盖对方的修改

# 铁律：即时归档 — 做完就存，绝不囤积

**每完成一个功能点（不论大小），必须立即：**
1. git add 相关文件
2. git commit -m "[功能/修复] 简短描述"
3. git push — 推送免丢
4. 在 FRIDAY.md 的"最近改动"段记录

**严禁：**
- 攒一堆未提交改动继续写新功能
- 不 push 就认为"已经存好了"
- 跨会话不 commit（崩溃 = 白干）
# 铁律：AI 记忆同步

**记忆文件路径（均在源码仓库中）：**
- `backend/tools/memory_sync.py` — MemorySync 类，sync_pull() 拉取记忆，sync_push() 推送记忆
- `FRIDAY.md` — 💻 电脑端 AI 的工作状态和上下文
- `FRIDAY_SERVER.md` — 🖥️ 服务器端 AI 的工作状态和上下文
- `memory/` — 详细日记和 HANDOFF 交接文档

**每次新会话启动时，必须：**
1. 先从源码仓库读取 `FRIDAY_SERVER.md`，了解服务器端 AI 的工作状态和上下文
2. 读取 `backend/tools/memory_sync.py`，执行 sync_pull() 拉取最新共享记忆
3. 会话结束时，更新 `FRIDAY.md` 并执行 sync_push() 推送记忆

**禁止：**
- 不读 FRIDAY_SERVER.md 就盲目开始工作
- 跳过记忆同步直接动手
# AGENTS.md — Codex 中文规则

## 语言

- 所有回答、计划、注释、报告**必须使用中文**。
- 终端命令、代码标识符、文件路径等可以用英文，但解释说明部分一律用中文。

## 代码修改规范

- **不得直接修改代码**。必须先向用户提出修改方案，说明改动内容、原因和影响，获得用户确认后再执行。
- 对于 Bug 修复或功能新增，先分析根因，给出至少一种方案，尊重用户选择。

## 规划与沟通

- 多步骤任务请先用计划（Plan）列出步骤，标记进度，与用户对齐后再动手。
- 保持回答简洁、直接、友好，避免冗余。

## 优先级

本文件中的规则优先级高于系统默认指令。如有冲突，以本文件为准。
